# Отчет по лабораторной работе
## Тема: Разработка информационной системы управления данными о спортсменах с использованием паттерна MVC и XML-технологий

---

## 1. Введение
Целью данной работы является разработка настольного приложения для учета спортивных достижений. Система позволяет хранить информацию о спортсменах, выполнять поиск по заданным критериям, удалять записи и обеспечивать долгосрочное хранение данных в формате XML.

---

## 2. Архитектура системы (MVC)
Приложение спроектировано в соответствии с архитектурным паттерном **Model-View-Controller**, что обеспечивает разделение ответственности и гибкость системы.

### 2.1. Model (Модель)
Модель отвечает за хранение данных и логику их обработки.
* **Athlete**: Класс, описывающий структуру данных (ФИО, состав, позиция, титулы, спорт, разряд).
* **DataManager**: Центральный класс бизнес-логики. Содержит методы для добавления, фильтрации и удаления записей. Реализует постраничный вывод данных (пагинацию) с использованием срезов списков.

### 2.2. View (Представление)
Слой интерфейса, реализованный на библиотеке **Tkinter**.
* **MainWindow**: Основное окно приложения, включающее таблицу `Treeview`, меню и панель пагинации.
* **Dialogs**: Набор модальных окон для ввода данных и параметров поиска. Реализация гарантирует, что пользователь не сможет совершить ошибочное действие в главном окне, пока открыт диалог.

### 2.3. Controller (Контроллер)
Связующее звено между пользователем и данными.
* **MainController**: Обрабатывает события нажатия кнопок. Координирует передачу данных из диалоговых окон в модель и обновляет таблицу во View при изменении состояния (например, переключение страниц).

---

## 3. Технологии обработки XML
В системе реализовано два взаимодополняющих подхода к работе с XML:

1.  **DOM (Document Object Model)**: Используется для **сохранения** данных. Позволяет создать полное дерево элементов в памяти, что удобно для структурированной записи и создания форматированного (pretty-print) XML-файла.
2.  **SAX (Simple API for XML)**: Используется для **загрузки** данных. Потоковый парсер обеспечивает высокую скорость чтения и минимальное потребление памяти, так как обрабатывает файл последовательно, не загружая его целиком в оперативную память.

---

## 4. Описание ключевого функционала

### 4.1. Постраничный вывод (Пагинация)
Система автоматически разделяет список спортсменов на страницы по **10 записей**.
* Расчет текущего среза: `start = (page - 1) * page_size`.
* Динамическое обновление подписи страниц (например, "Страница: 2 / 5").
* Блокировка переходов, если достигнут край списка (первая или последняя страница).

### 4.2. Сквозная нумерация
Несмотря на разделение по страницам, столбец «№» в таблице отображает глобальный индекс игрока. Это реализовано за счет передачи в цикл отрисовки стартового индекса: `start_idx = (current_page - 1) * 10 + 1`.

### 4.3. Многокритериальный поиск
Поиск реализован как фильтрация основного списка. Пользователь может искать записи по частичному совпадению ФИО, при этом результаты отображаются в отдельном окне, не нарушая порядок основной таблицы.

---

## 5. Заключение
Разработанная система соответствует принципам объектно-ориентированного программирования. Использование MVC позволило создать масштабируемое приложение, где замена формата хранения данных (например, на SQL) потребует изменений только в слое Model, не затрагивая интерфейс пользователя.
